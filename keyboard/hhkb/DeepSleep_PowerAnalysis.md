# HHKB Deep Sleep 전력 분석 보고서

## 요약

현재 HHKB의 Deep Sleep 구현은 실제로 전력을 절약하지 못하고 있습니다. 일반 블루투스 모드와 Deep Sleep 모드의 배터리 지속 시간이 거의 동일한 이유는 다음과 같습니다:

- **MCU가 Sleep 모드를 사용하지 않음** (10-15mA 지속 소비)
- **RN-42 블루투스가 활성 상태 유지** (25-40mA 지속 소비)
- **매트릭스 스캔이 계속 동작** (1-2mA 지속 소비)
- **유일한 절약은 LED 끄기** (5-10mA 절약)

## 전력 소비 비교

| 구성 요소 | 일반 모드 | Deep Sleep 모드 | 실제 절약 |
|-----------|-----------|-----------------|-----------|
| ATmega32U4 MCU | 10-15mA | 10-15mA | 0mA |
| RN-42 블루투스 | 25-40mA | 25-40mA | 0mA |
| 매트릭스 스캔 | 1-2mA | 1-2mA | 0mA |
| LED | 5-10mA | 0mA | 5-10mA |
| **총 소비량** | **40-70mA** | **35-60mA** | **~10%** |

## 근본적인 문제점

### 1. ATmega32U4 Sleep 모드 미사용

현재 코드는 MCU의 Sleep 모드를 전혀 사용하지 않습니다:

```c
// power_management.c:288-297
static void power_mgr_execute_sleep(void) {
    /* HHKB는 매트릭스 스캔 방식이므로 Sleep 모드 사용 안함 */
    /* 실제 Sleep 모드는 매트릭스 스캔과 호환되지 않음 */
    // 아무것도 하지 않고 리턴
}
```

ATmega32U4의 Power-down 모드를 사용하면 <0.1mA까지 전력 소비를 줄일 수 있지만, 매트릭스 스캔과 호환되지 않아 사용하지 못하고 있습니다.

### 2. 매트릭스 스캔 지속

Deep Sleep 중에도 10ms마다 전체 키보드를 스캔합니다:

```c
// main.c:197-227
if (is_sleeping) {
    /* Deep Sleep 상태에서도 매트릭스 스캔 (Wake-up 감지용) */
    matrix_scan();  // 전력 소비 지속
    
    // Enter 키 체크...
    wait_ms(10);
    continue;
}
```

### 3. RN-42 블루투스 모듈 활성 상태

RN-42에 실제 Sleep 명령을 보내지 않습니다:

```c
// rn42_sleep.c:61-73
bool rn42_enter_deep_sleep(void) {
    /* RN-42 모듈과 상호작용하지 않고 단순히 상태만 변경 */
    current_state = RN42_STATE_DEEP_SLEEP;
    return true;
}
```

## 아키텍처 한계

### 매트릭스 스캔 방식의 한계

HHKB는 8x8 매트릭스 방식으로 키를 감지합니다:
- 각 키는 행(row)과 열(column)의 교차점
- Wake-up을 위해서는 지속적인 스캔 필요
- Sleep 모드 진입 시 스캔 불가능

### 인터럽트 핀 부족

ATmega32U4의 외부 인터럽트 핀은 제한적입니다:
- INT0-INT3: 4개
- PCINT0-PCINT7: 8개
- 64개 키를 모두 인터럽트로 감지 불가능

## 해결 방안

### 단기적 개선 (즉시 적용 가능)

1. **매트릭스 스캔 주기 연장**
   - 10ms → 100-500ms로 변경
   - 예상 절약: 1-2mA → 0.1-0.2mA

2. **CPU 클럭 감소**
   - 16MHz → 1MHz로 동적 변경
   - 예상 절약: 10-15mA → 2-3mA

### 중기적 개선 (구현 복잡도 중간)

1. **WDT(Watchdog Timer) 기반 Wake-up**
   - Power-down 모드 + WDT 인터럽트
   - 주기적으로 깨어나서 Enter 키만 체크
   - 예상 절약: 10-15mA → <1mA

2. **RN-42 Sniff 모드 활용**
   - 블루투스 연결 유지하며 전력 절약
   - 예상 절약: 25-40mA → 5-10mA

### 장기적 개선 (하드웨어 변경 필요)

1. **Wake-up 전용 회로 추가**
   - 각 행(row)에 OR 게이트 추가
   - 하나의 인터럽트 핀으로 모든 키 감지
   - 진정한 Deep Sleep 가능

2. **전용 Wake-up 키 지정**
   - Enter 키를 외부 인터럽트 핀에 직접 연결
   - 매트릭스 스캔 없이 Wake-up 가능

## 권장 구현 순서

1. **Phase 1**: 매트릭스 스캔 주기 연장 (즉시)
2. **Phase 2**: WDT 기반 Power-down 모드 (1주일)
3. **Phase 3**: RN-42 Sniff 모드 통합 (2주일)
4. **Phase 4**: 하드웨어 개선 검토 (장기)

## 예상 결과

현재 구현으로는 약 10% 정도의 전력만 절약되지만, 권장 개선안을 적용하면:

| 구현 단계 | 예상 소비 전력 | 배터리 지속 시간 |
|----------|---------------|-----------------|
| 현재 | 35-60mA | ~24시간 |
| Phase 1 | 30-55mA | ~30시간 |
| Phase 2 | 15-40mA | ~60시간 |
| Phase 3 | 5-15mA | ~200시간 |
| Phase 4 | <1mA | >1000시간 |

## 결론

현재의 "Deep Sleep"은 이름뿐인 기능으로, 실제 전력 절약 효과가 미미합니다. 매트릭스 스캔 방식의 키보드에서 진정한 Deep Sleep을 구현하려면 하드웨어 지원이나 창의적인 소프트웨어 해결책이 필요합니다.
